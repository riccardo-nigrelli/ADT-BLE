ADT-BLE

An integrable Python library — async + sync — and a small CLI for talking to Additel calibrators over Bluetooth Low Energy. Primary target: the ADT226 (works with the ADT227 and …Ex variants).

Python 3.8+ macOS · Windows · Linux Bleak Typer CLI MIT
Get started → GitHub ↗

The library handles the fiddly parts of the Additel BLE protocol for you — connection, GATT characteristic resolution, the CODE? → @ handshake, fragmented-notification reassembly and automatic error-queue checking — behind a plain query() / write() API with typed exceptions.

Async & syncUse async/await or a plain blocking API.
Automatic handshakeAnswers the device's CODE? so commands work.
UUID auto-discoveryFinds the right characteristic per model.
Error checkingReads SYSTem:ERRor? and raises typed errors.
Robust framingReassembles fragmented notifications.
One dependencyJust bleak; fully typed.

Installation

Requirements: Python 3.8+, Bluetooth enabled, and the calibrator powered on and in range.

As a library, to integrate into another program:

# from a local clone
pip install .
# or straight from GitHub
pip install "git+https://github.com/riccardo-nigrelli/ADT-BLE.git"

With the test CLI (the cli extra):

pip install ".[cli]"

Development environment (editable + CLI + tests):

python -m venv .venv
# macOS/Linux: source .venv/bin/activate
# Windows:   .venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt

Quickstart

Async — recommended

import asyncio
from additel_ble import AdditelBLE

async def main():
    async with AdditelBLE(name="ADT226") as dev:   # connect + handshake
        print(await dev.identify())          # *IDN?
        print(await dev.measure())           # MEASure:CH? PV
        print(await dev.query("CALibrator:MEASure:PRESsure:UNIT?"))

asyncio.run(main())

Synchronous — for non-async programs

from additel_ble import AdditelBLESync

with AdditelBLESync(name="ADT226") as dev:
    print(dev.identify())
    print(dev.measure())

CLI

adt-ble scan                          # list nearby BLE devices
adt-ble send --name ADT226 "*IDN?"     # connect, send, disconnect
Connect by name (scanned for) or address (direct). One of them is required — there is no default device name.

How it works

  1. Transport — UART over GATT. Commands are written to a write characteristic; replies arrive as notifications.
  2. Handshake. After subscribing, the device sends CODE?. The client must answer @\r\n within ~5 s, or the device disconnects and ignores every command. The library does this automatically (handshake="@").
  3. Characteristic resolution. Explicit overrides → Additel's documented UUIDs → auto-discovery of a single characteristic that supports both notify and write.
  4. SCPI commands terminated with \r\n; replies (which may be fragmented) are buffered until the terminator.
  5. Error checking. With check_errors=True (default), the library reads SYSTem:ERRor? after each command and raises DeviceCommandError on a non-zero code. The queue is cleared on connect and popped on each read.
The generic 226 guide says you can send commands right after CODE? without replying — that is incomplete. Without the @ handshake the device stays silent and disconnects after ~5 s.

AdditelBLESync(...)

Same constructor and methods as AdditelBLE, but blocking: it runs an asyncio loop on a background thread and marshals each call onto it, keeping a single stateful connection open. Supports with.

from additel_ble import AdditelBLESync
with AdditelBLESync(address="AA:BB:CC:DD:EE:FF") as dev:
    print(dev.query("SYSTem:VERSion?"))

Exceptions

All inherit from AdditelError.

ExceptionRaised when
DeviceNotFoundErrorNo BLE device matched the name during a scan.
ConnectionFailedErrorThe BLE connection could not be established.
CharacteristicNotFoundErrorNo usable notify/write characteristic was found.
CommandTimeoutErrorNo reply arrived within the timeout.
DeviceCommandErrorThe device queued an error — exposes .code and .message.
 
from additel_ble import AdditelBLE, DeviceCommandError, CommandTimeoutError

async with AdditelBLE(name="ADT226") as dev:
    try:
        value = await dev.query("MEASure:CH? PV")
    except DeviceCommandError as e:
        print("device rejected it:", e.code, e.message)
    except CommandTimeoutError:
        print("no reply")

CLI

Installed with the cli extra as adt-ble (also runnable as python -m additel_ble.cli).

CommandDescription
adt-ble scan [--name T] [--timeout N]List nearby BLE devices.
adt-ble uuid <NAME> [--all]Print the address/UUID for a device name.
adt-ble send [CMDS…] (--name T | --uuid ADDR)Connect, send command(s), print replies, disconnect.
 
adt-ble send --name ADT226 "*IDN?"
adt-ble send --uuid AA:BB:CC:DD:EE:FF "*IDN?" "MEASure:CH? PV"   # several commands, one connection
adt-ble send -n ADT226 -v "*IDN?"                    # -v: GATT table + debug logs
adt-ble send -n ADT226 --no-error-check "…"          # skip the SYSTem:ERRor? check
All commands passed to a single send share one connection and one handshake, so it is faster than running send repeatedly.

Commands & errors

*IDN? is an IEEE 488.2 common command: no parameter, no mode — it always answers. Most other commands do not answer unless used correctly:

  • Missing parameter. Read the value with MEASure:CH? PV (the PV parameter is required). MEASure:CH? alone returns nothing.
  • Wrong mode. CALibrator:… commands only reply in Calibrator mode.
  • Set commands (e.g. …:UNIT <id>) legitimately return nothing.

When a command is invalid the device stays silent and queues an error. The library reads it and raises DeviceCommandError; you can also read it manually with await dev.error(). Mode-independent commands that always reply (handy for a smoke test): SYSTem:VERSion?, SYSTem:DATE?, SYSTem:BLUEtooth:NAMe, SYSTem:BATTery:CAPacity?.

Handshake & UUIDsDocumented in Additel's Bluetooth Protocol for the ADT685 (same UUIDs as the 226/227) — see the PDF and the notes.
Full command setThe complete SCPI reference is in scpi_commands.md and 226_227_commands.txt.

Troubleshooting

SymptomCause / fix
No device foundDevice off/out of range, Bluetooth off, or (macOS) permission not granted. Try adt-ble scan.
Connects, but every command times outKeep the handshake on (handshake="@"). Check the ready state with -v.
One command returns no replyMissing parameter or wrong mode — read dev.error().
DeviceCommandError … -109The device reported a command error (here: missing parameter).
Wrong characteristic chosenPass notify_uuid=/write_uuid= from the -v GATT table.
